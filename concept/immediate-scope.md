# Immediate Scope

## Purpose of the first release

The first public release of Two-Set Policy will establish a generic research framework for creating a traceable operational policy layer from an existing human-readable policy.

The first release will not attempt to automate legal interpretation or create a production policy-enforcement platform.

Its purpose is to define the minimum concepts, artefacts, workflows, and safeguards required before software systems or AI agents can be expected to implement, monitor, or enforce policy.

## Immediate objective

The immediate objective is to define a reusable method for:

1. extracting policy structure
2. identifying actors and authorities
3. identifying obligations, permissions, and prohibitions
4. detecting interpretation vulnerabilities
5. detecting monitoring-system vulnerabilities
6. modelling implementation sequences
7. identifying redressal gaps
8. drafting a provisional operational policy
9. tracing operational rules to source policy
10. recording unresolved policy dependencies

## Included in the initial scope

The initial framework will include:

- policy purpose and scope
- jurisdiction and legal status
- rights holders
- duty bearers
- action owners
- decision authorities
- obligations
- permissions
- prohibitions
- triggers
- evidence requirements
- exceptions
- human-review conditions
- escalation paths
- monitoring requirements
- notification duties
- redressal mechanisms
- audit requirements
- change-control authority
- unresolved questions
- source traceability

## Initial outputs

The first release will aim to produce:

- policy context JSON
- actor and authority map
- obligation and prohibition register
- interpretation-vulnerability register
- monitoring-vulnerability register
- redressal gap register
- implementation sequence diagrams
- provisional operational policy JSON
- unresolved TODO register
- traceability matrix

## Initial policy case

The National Digital Health Blueprint of India will be used as Case 001.

The first analysis will be limited to selected workflows such as:

- consented health-record sharing
- consent withdrawal
- emergency access
- identity uncertainty
- research and analytics access
- monitoring
- correction and redressal

## Explicitly outside the initial scope

The first release will not provide:

- automatic legal interpretation
- production system enforcement
- certified regulatory compliance
- a universal policy ontology
- automated resolution of normative questions
- automated redressal decisions
- cross-jurisdiction policy harmonisation
- a production-grade policy engine
- proof that the framework applies to every policy domain
- an official interpretation of any policy document

## Initial technology approach

The first release will use a lightweight, human-reviewed technology approach based on:

- Markdown
- JSON
- JSON Schema
- CSV
- Mermaid diagrams
- Git version history
- manual and AI-assisted extraction
- human validation

A database, runtime rule engine, and production application are not part of the initial release.

## Success criteria for the first release

The first release will be considered useful if it demonstrates that:

- the framework can be applied to a real policy
- operational ambiguities can be made visible
- action owners and authorities can be identified
- redressal requirements can be represented
- monitoring risks can be distinguished from policy compliance
- unresolved questions remain visible
- every proposed operational rule can be traced to source policy
